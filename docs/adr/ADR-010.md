# ADR-010: Prevent Accidental Hard Deletion of Questions

## Status

Accepted

## Context

In Sprint C.1, the historical assessment data loss vulnerability was mitigated by introducing a soft-delete mechanism on the `Question` model using the `is_active` flag.

Questions are now archived via:

```python
question.archive()
```

instead of being physically removed from the database.

However, the model still permits direct execution of:

```python
question.delete()
```

which will trigger Django's cascade deletion behavior and permanently remove related historical records such as:

* AssessmentQuestion
* StudentAnswer

Although normal application workflows no longer use physical deletion, accidental usage by developers, admin scripts, management commands, or future features remains possible.

## Decision

We will enforce protection against physical deletion of historical question data at the model layer.

The `Question.delete()` method will be overridden to block physical deletion in two main scenarios:
1. **Archived status**: Archived questions (`is_active=False`) are considered historical records and must never be physically deleted through normal application code, regardless of whether any active database relationships exist.
2. **Historical relations**: Active questions with historical assessment links (`AssessmentQuestion` through-model relations) or student attempt history (`StudentAnswer` records) must never be physically deleted.

We choose to raise `django.db.models.ProtectedError` instead of `ValidationError` for these checks:
- **Standard Django Behavior**: `ProtectedError` is Django's native exception for referential integrity protection.
- **Admin Panel Integration**: The Django Admin panel gracefully catches `ProtectedError` and presents a clear, user-friendly dependency warning UI.
- **Developer Intent**: It immediately signals to developers that this is a hard database dependency and referential integrity protection, rather than a transient form validation error.

```python
def delete(self, *args, **kwargs):
    if not self.is_active:
        raise ProtectedError(
            "Archived questions cannot be deleted.",
            [self],
        )
    if self.assessmentquestion_set.exists() or self.studentanswer_set.exists():
        raise ProtectedError(
            "Questions with assessment history cannot be deleted.",
            [self],
        )
    super().delete(*args, **kwargs)
```

Active, unused draft questions that have no assessment links or student answers can still be safely deleted (e.g., during cleanups or authoring phases).

## Consequences

### Positive

* Eliminates accidental historical data loss.
* Enforces a single deletion policy across the codebase.
* Makes destructive operations explicit and auditable.
* Protects future developers from unintentionally triggering cascades.

### Negative

* Requires a dedicated administrative deletion path.
* Existing scripts that rely on `question.delete()` must be updated.

## Alternatives Considered

### Option A — Keep Current Soft Delete Only

Pros:

* No additional implementation effort.

Cons:

* Accidental hard deletes remain possible.

Rejected.

### Option B — Override delete() and Require Service-Level Hard Deletes

Pros:

* Strongest protection against developer mistakes.
* Consistent with historical data preservation requirements.

Accepted.

### Option C — Introduce Third-Party Soft Delete Framework

Example:

```python
django-safedelete
```

Pros:

* Feature-rich ecosystem.

Cons:

* Additional dependency.
* Unnecessary complexity for current project scale.

Rejected.

## Target Sprint

Sprint C.2 or Sprint C.3
