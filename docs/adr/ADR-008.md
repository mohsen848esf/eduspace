# ADR-008: Student Live Session Auto Refresh

## Status

Accepted (Deferred)

## Context

Why this issue exists and where it was discovered.
This issue was identified during the Sprint B.7 Frontend Architecture Review. Currently, `useSessions()` is only queried when the component mounts or when manually invalidated. Consequently, a student who keeps the Dashboard open will not reactively see a new live session started by the host until they manually refresh the page or perform tab navigation.

## Decision

What architectural decision is proposed.
Enable background polling specifically for students on active live sessions, or integrate active WebSockets/Server-Sent Events (SSE) notification streams.
For students, the query would run as:
```typescript
useQuery({
  queryKey: ["sessions", { status: "live" }],
  queryFn: fetchSessions,
  refetchInterval: 30000,
});
```
Currently, this is deferred as it has no security or data integrity impact, does not affect backend systems, and solely serves to improve UX.

## Consequences

Benefits, risks, trade-offs.
- **Benefits**:
  - Dramatically improves user experience (UX) for students, allowing them to join live classes without needing manual refreshes.
- **Risks & Trade-offs**:
  - Adds minor query load on the backend for polling, which can scale up with the number of concurrent active student sessions.

## Target Sprint

Sprint D or Sprint E.
