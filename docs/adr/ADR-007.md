# ADR-007: Remove Legacy AcademyClass.room

## Status

Proposed

## Context

Why this issue exists and where it was discovered.
During Sprint B, the application migrated room relationships from `AcademyClass -> room` to `AcademyClass -> Session -> active_room`. To prevent breaking active client setups and ensure smooth rollout, the database column `AcademyClass.room` was kept, and a backward compatibility sync layer was implemented (Task B.4) so that modifications to a session's active room are mirrored back to `AcademyClass.room`. Once all clients (frontend web app and mobile/legacy integrations) migrate to the new Session-based rooms model, this compatibility layer will be redundant and should be purged.

## Decision

What architectural decision is proposed.
Permanently remove the `room` column from `AcademyClass` (via a standard Django database migration) and delete the corresponding synchronization logic inside `Session.save()` and `AcademyClassSerializer`.

## Consequences

Benefits, risks, trade-offs.
- **Benefits**:
  - Simplifies the domain schema and resolves the circular/redundant database relationships.
  - Eliminates synchronization overhead, reducing DB writes and potential transaction deadlocks.
  - Aligns database state exactly with the logical architecture.
- **Risks & Trade-offs**:
  - A breaking database change: any legacy/third-party clients that still access or modify the `room` field on `AcademyClass` will fail.
  - Requires a planned deprecation window and release cutoff.

## Target Sprint

Post-Migration Cleanup Sprint.
