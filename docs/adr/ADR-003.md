# ADR-003: Move Compatibility Synchronization Out of Session.save()

## Status

Proposed

## Context

Why this issue exists and where it was discovered.
To maintain backward compatibility during the Sprint B refactor, `Session.save()` was overridden to automatically update the legacy `AcademyClass.room` field whenever `active_room` changes (Task B.4). However, keeping business and integration logic inside model `save()` methods is an anti-pattern as it couples low-level database operations with high-level business sync rules. This causes unintended side effects during bulk updates or migrations.

## Decision

What architectural decision is proposed.
Refactor the synchronization logic out of `Session.save()` and move it into an explicit service function in `SessionService` (or a dedicated post-save signal receiver). This logic should be kept clearly scoped until the legacy room column is fully deleted (as planned in ADR-002).

## Consequences

Benefits, risks, trade-offs.
- **Benefits**:
  - Decouples the database persistence layer (`models.py`) from business synchronization rules.
  - Prevents unexpected side effects when saving/creating session models from outside the standard workflows (e.g., in shell scripts, migrations, or bulk commands).
  - Keeps the models clean and focused.
- **Risks & Trade-offs**:
  - If a session is modified bypassing the service layer (e.g. standard Django Admin panel saving), the compatibility synchronization will not execute. This risk is minimized by wrapping admin operations or adding a signal handler.
