# ADR-006: Celery Attendance Processing

## Status

Proposed

## Context

Why this issue exists and where it was discovered.
Currently, attendance generation runs through a transaction `on_commit` callback (`transaction.on_commit()`) inside `SessionService` on the backend. This was introduced during Task B.5 as a simple and lightweight mechanism to auto-populate student attendance metrics after a live session ends. However, because it runs inside the application process synchronously following the transaction commit, any server process crash, restart, or network interruption immediately after the database commit will result in a completed session without its corresponding student attendance logs.

## Decision

What architectural decision is proposed.
Replace the in-process post-commit execution with a durable Celery task queue using Redis as a broker and result backend. The task will handle student connection calculations asynchronously with robust retry capabilities.

## Consequences

Benefits, risks, trade-offs.
- **Benefits**:
  - Resilient to worker and server crashes: task states are preserved in the broker.
  - Performance: unblocks the web application processes by offloading heavy parsing work to Celery background workers.
  - Scalability: queue workers can be scaled out independently as class and student attendance volume grows.
- **Risks & Trade-offs**:
  - Introduces Celery and Redis as operational dependencies, increasing setup complexity.
  - Eventual consistency: student attendance updates are no longer immediately visible in the database right when the transaction commits.
