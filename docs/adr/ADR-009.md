# ADR-009: Historical Assessment Data Preservation

## Status

Accepted

## Context

During the review of Task C.1 (Assessment Models), an issue was identified regarding the physical deletion of questions. Deleting a `Question` would cascade (via Django's `on_delete=models.CASCADE` on `AssessmentQuestion` and `StudentAnswer`) and delete all related historical assessment records and student submission answers. This results in irreversible loss of assessment history, grading logs, and students' submitted work, violating strict requirements for immutable historical reporting and auditing.

## Decision

We have chosen a **Soft Delete / Archive** strategy for the `Question` model:
1. Add an archival field `is_active = models.BooleanField(default=True)` to the `Question` model.
2. Provide helper methods `archive()` and `restore()` on the model to safely change the active state without executing a database-level `DELETE`.
3. Keep database relationships as `on_delete=models.CASCADE` for physical database-level consistency under rare, administrative hard-deletions, but never perform physical deletion through normal application workflows.
4. Implement a custom manager (`QuestionManager`) and query set (`QuestionQuerySet`) that exposes:
   - `Question.objects.active()` for normal application flows (e.g. creating new assessments, listing active questions).
   - `Question.objects.with_archived()` (which is equivalent to the default queryset) to ensure that existing assessments, submissions, and student answers continue to resolve and load historical questions.
5. In Django, lookups and relations (`assessment.questions.all()`) depend on the model's default manager. Therefore, the default manager `objects` must continue to return both active and archived questions to prevent historical assessments from losing their linked questions.

## Consequences

- **Benefits**:
  - **Zero Data Loss**: Archiving a question leaves its database row fully intact, preserving all historical `AssessmentQuestion` and `StudentAnswer` records.
  - **No Cascading Deletion**: Normal application workflows perform soft deletes (marking `is_active=False`), which avoids triggering Django's `CASCADE` delete on related objects.
  - **Auditability**: Historical grades, exam submissions, and answers remain completely queryable and unchanged forever.
- **Risks & Trade-offs**:
  - Developers must use `Question.objects.active()` explicitly when querying active questions in standard workflows (e.g., when adding questions to new assessments) to prevent archived questions from showing up.
  - Using `is_active=True` adds a small cognitive overhead compared to standard physical deletions, but ensures total history preservation.
